# Развёртывание на сервере (production)

## 1. Требования к серверу

- Ubuntu Server 22.04/24.04 LTS, 2+ vCPU, 4+ ГБ RAM, 40+ ГБ SSD (см. рекомендации в чате — 4 vCPU/8 ГБ с запасом на рост).
- Домен, A-записью указывающий на IP сервера (нужен для HTTPS).
- Открытые порты 80 и 443 (443 — обязательно наружу, 80 — для ACME-проверки и редиректа).

## 2. Установка Docker

```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
# перелогиньтесь, чтобы группа docker применилась
```

## 3. Копирование проекта на сервер

```bash
git clone <ваш-репозиторий> crm && cd crm
# либо просто скопируйте директорию проекта через scp/rsync
```

## 4. Настройка переменных окружения

```bash
cp .env.production.example .env
nano .env
```

Обязательно замените:
- `POSTGRES_PASSWORD`, `SEED_ADMIN_PASSWORD` (и `SEED_SUPERADMIN_PASSWORD`, если заводите суперадминистратора — см. шаг 6) — сгенерируйте: `openssl rand -hex 24`. Пароли короче 16 символов или содержащие `CHANGE_ME` backend откажется использовать при бутстрапе в production.
- `JWT_SECRET` — сгенерируйте: `openssl rand -hex 32`. Backend не запустится в production, если секрет короче 32 байт или похож на плейсхолдер.
- `DOMAIN` — ваш реальный домен (A-запись уже должна на него указывать)
- `LETSENCRYPT_EMAIL` — для уведомлений об истечении сертификата

`openssl rand -base64 ...` тоже подходит по длине, но `-hex` безопаснее вставлять прямо в `DATABASE_URL`: base64-алфавит может дать `/`, который ломает разбор URI.

## 5. Получение HTTPS-сертификата (один раз)

```bash
chmod +x deploy/init-letsencrypt.sh scripts/*.sh
./deploy/init-letsencrypt.sh
```

Скрипт поднимет nginx с временным сертификатом, получит настоящий от Let's Encrypt и перезагрузит nginx.
Скрипт идемпотентен — повторный запуск при уже действующем сертификате ничего не трогает.

Продление — отдельная задача на хосте (не выполняется автоматически ничем внутри Docker):

```bash
crontab -e
```
Добавьте (дважды в сутки, как рекомендует Let's Encrypt):
```
0 3,15 * * * /path/to/crm/scripts/renew-certificates.sh >> /var/log/crm-cert-renew.log 2>&1
```
Продление само по себе почти всегда no-op (сертификат обновляется только за ~30 дней до истечения),
но скрипт каждый раз перезагружает nginx, чтобы гарантированно подхватить новый сертификат, когда
продление всё же произошло.

## 6. Запуск всего стека

```bash
docker compose -f docker-compose.prod.yml up -d --build
docker compose -f docker-compose.prod.yml logs -f backend
```

При первом запуске backend применит миграции Prisma и запустит сервер — но администратор ещё не создан. Заведение первого аккаунта — отдельный, однократный шаг (сознательно не выполняется автоматически при каждом старте контейнера, чтобы забытая/пустая переменная в `.env` не привела к тому, что в интернете окажется доступен аккаунт с известным паролем):

```bash
docker compose -f docker-compose.prod.yml run --rm backend npm run seed
```

Это создаст администратора из `SEED_ADMIN_USERNAME` / `SEED_ADMIN_PASSWORD`. **Смените этот пароль сразу после первого входа.**

Если нужны несколько филиалов: задайте `SEED_SUPERADMIN_USERNAME` / `SEED_SUPERADMIN_PASSWORD` в `.env` до запуска команды выше — так же создастся суперадминистратор (видит и создаёт филиалы, назначает первого админа каждому). Обычные администраторы такую роль назначить не могут — это единственный способ завести первого суперадминистратора. При повторном запуске команды уже существующие учётные записи не трогаются (seed идемпотентен) — её можно безопасно повторить, если нужно досоздать суперадминистратора позже.

Откройте `https://<ваш-домен>` — должна открыться страница входа.

## 7. Бэкапы базы данных

```bash
crontab -e
```
Добавьте (ежедневный бэкап в 03:00, хранить 14 дней):
```
0 3 * * * /path/to/crm/scripts/backup-db.sh >> /var/log/crm-backup.log 2>&1
```

Восстановление из бэкапа: `scripts/restore-db.sh backups/crm_<дата>.dump`. Скрипт сначала
восстанавливает архив во временную БД и проверяет её, и только потом подменяет рабочую — сбой
на любом шаге не трогает уже работающие данные.

## 8. Обновление после изменений в коде

```bash
git pull
docker compose -f docker-compose.prod.yml up -d --build
```

Контейнеры без простоя не гарантируются (кратковременный рестарт backend/frontend), но БД и её том не трогаются.

## 9. Что уже учтено в конфигурации

- `db` и `backend` не публикуют порты наружу — доступны только внутри docker-сети.
- Frontend отдаёт статику и проксирует `/api/*` на backend через nginx — CORS не нужен, всё на одном домене.
- Секреты (`JWT_SECRET`, пароли) берутся только из `.env`, которого нет в git (см. `.gitignore`).
- security-заголовки (`X-Frame-Options`, `X-Content-Type-Options`), редирект HTTP→HTTPS.
- Продление сертификата и бэкап БД — cron-задачи на хосте (шаги 5 и 7 этого документа), не выполняются
  автоматически сами по себе после разворачивания.

## 10. Чего в этой конфигурации нет (сделать по мере роста)

- Централизованные логи/мониторинг (например, добавить Prometheus/Grafana или внешний сервис).
- Rate limiting на `/api/auth/login` (защита от подбора пароля) — стоит добавить в nginx (`limit_req`) при выходе в публичный доступ.
- Горизонтальное масштабирование backend (сейчас один контейнер).
